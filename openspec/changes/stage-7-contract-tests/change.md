# Тестирование контрактов Quantower (Этап 7)

## Зачем

Во время тестового запуска на реальном счете был обнаружен баг: бот "завис", ожидая ответа от биржи при нулевом балансе, так как `QuantowerAdapter` не обработал синхронный отказ метода `PlaceOrder`. Для построения надежной системы авто-перезапуска и восстановления (Reconciliation) нам необходимо точно знать, как API Quantower реагирует на сбои, отказы, Rate Limits и потерю связи.

## Требуемое поведение

- Созданы автоматизированные или задокументированные ручные тесты в проекте `DCAnt2.QuantowerTests` для проверки фактического поведения API брокера.
- Подтверждено, обеспечивает ли `ClientRequestId` (поле Comment) уникальность и защиту от двойной отправки, а также возможность поиска ордера после рестарта терминала.
- Определено поведение API при таймаутах и ошибках валидации (например, нехватка средств).
- Исправлена логика `QuantowerAdapter.ProcessOutbox`: метод `PlaceOrder` теперь проверяет возвращаемое значение (синхронный отказ) и генерирует `RejectionMessage`, если заявка отклонена биржей сразу.
- Результаты исследования задокументированы в markdown-файл или детальные комментарии, чтобы на них можно было опираться при разработке Этапа 11 (Сверка и Восстановление).

## Не входит

- Разработка самой системы восстановления после сбоев (Recovery).
- Разработка авто-перезапуска цикла (Restart с активной сеткой) — это относится к Этапу 9.
- Внесение изменений в математику DCA.
