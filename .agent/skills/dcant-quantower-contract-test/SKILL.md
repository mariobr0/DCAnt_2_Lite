# DCAnt Quantower Contract Test

## Назначение

Определять фактическое поведение Quantower и конкретного broker connector экспериментально.

Contract test отвечает на вопрос:

> Что Quantower и брокер действительно делают в конкретном сценарии?

Contract test не должен доказывать правильность торговой стратегии.

## Главный принцип

Документация, успешный API-вызов и сообщение `success` не являются достаточным доказательством фактического результата на бирже.

Нужно отдельно проверить:

- локальный ответ API;
- события Quantower;
- состояние `Core.Orders`;
- broker order history;
- Executions;
- фактическую позицию.

## Перед началом

1. Прочитай конкретный вопрос или предположение.
2. Прочитай только релевантный код Quantower Adapter.
3. Определи:
   - connector;
   - тип счета;
   - instrument;
   - position mode;
   - направление;
   - тип ордера.
4. Убедись, что испытание проводится на демо-счете.
5. Не объединяй несколько независимых гипотез в один тест.

## Структура contract test

Каждый тест должен содержать:

### Contract ID

Короткий стабильный идентификатор.

Пример:

`QT-CANCEL-001`

### Вопрос

Что именно требуется установить.

Пример:

> Когда Quantower считает отмену limit-order подтвержденной?

### Предусловия

Указать:

- версию Quantower;
- connector;
- account type;
- symbol;
- position mode;
- состояние соединения;
- исходные ордера и позицию.

### Действие

Одна конкретная операция:

- `PlaceOrder`;
- `CancelOrder`;
- `ModifyOrder`;
- `ClosePosition`;
- reconnect;
- restart.

### Наблюдения

Записать:

- время операции;
- локальный результат вызова;
- ID запроса;
- broker order ID;
- callbacks;
- изменение `Core.Orders`;
- Executions;
- изменение позиции;
- состояние на стороне брокера.

### Критерий подтверждения

Заранее указать, какие наблюдения доказывают контракт.

### Результат

Один из вариантов:

- `Confirmed`;
- `Rejected`;
- `ConnectorSpecific`;
- `Inconclusive`;
- `NotSupported`.

## Обязательные группы тестов

### 1. Идентификаторы

Проверить:

- сохраняется ли `ClientRequestId`;
- максимальную длину;
- разрешенные символы;
- изменяется ли ID после modify;
- можно ли найти ордер после reconnect;
- уникален ли `ExecutionId`;
- переиспользуется ли `ExecutionId` в разные дни.

### 2. PlaceOrder

Проверить:

- что означает успешный ответ;
- когда ордер появляется в `Core.Orders`;
- порядок callbacks;
- поведение при timeout;
- поведение при rejection;
- возможность появления Execution до OrderAccepted.

### 3. Partial fills

Проверить:

- число событий;
- изменение filled quantity;
- наличие отдельных Executions;
- порядок Execution и order update;
- обновление position quantity;
- duplicate callbacks.

### 4. Cancel

Проверить:

- что означает успешный ответ;
- когда ордер исчезает из active orders;
- какое событие подтверждает отмену;
- cancel/fill race;
- repeated cancel;
- timeout отмены;
- состояние после reconnect.

### 5. Modify

Проверить:

- сохраняется ли broker order ID;
- создается ли новый ордер;
- сохраняется ли Comment;
- сохраняется ли ClientRequestId;
- отменяется ли старый ордер;
- возможно ли одновременное существование старого и нового;
- что означает `Modify succeeded`;
- как modify взаимодействует с OCO и reduce-only.

Не разрешать использование `ModifyOrder` в критической логике до подтверждения необходимого контракта.

### 6. TP и SL

Проверить:

- reduce-only;
- position-scoped orders;
- stop-market;
- partial position close;
- изменение quantity;
- одновременное исполнение TP и SL;
- удаление защиты после закрытия позиции;
- ручное удаление TP или SL.

### 7. Reconnect и restart

Проверить:

- восстановление active orders;
- восстановление Comments и IDs;
- доступность recent Executions;
- глубину истории;
- порядок событий после reconnect;
- delayed callbacks;
- несоответствие локальной и broker position.

### 8. Deployment

Проверить:

- копирование ProjectReference DLL;
- копирование NuGet dependencies;
- загрузку native SQLite runtime;
- Release build;
- отсутствие старых DLL;
- совместимость deployment set.

## Требования к логированию теста

Каждая запись должна по возможности содержать:

- UTC timestamp;
- contract ID;
- strategy instance ID;
- local request ID;
- client request ID;
- broker order ID;
- execution ID;
- event name;
- order status;
- quantity;
- price;
- comment;
- position quantity.

Не использовать только человекочитаемую строку без идентификаторов.

## Анализ результата

Строго разделяй:

### Подтвержденный факт

Непосредственно наблюдаемое поведение.

### Вероятная интерпретация

Объяснение факта, которое еще может иметь альтернативы.

### Неизвестно

Данные, которых недостаточно.

### Архитектурное последствие

Как результат влияет на DCAnt.

Пример:

> `ModifyOrder` создает новый broker order ID и не сохраняет Comment.  
> Следствие: ownership нельзя восстанавливать только по Comment, а замена TP через modify требует отдельной correlation-модели.

## Формат отчета

### Коротко

Какой контракт проверялся и что установлено.

### Условия теста

Версия, connector, account, symbol и режим.

### Хронология

События в точном временном порядке.

### Подтвержденные факты

Только наблюдаемые результаты.

### Вывод

`Confirmed`, `Rejected`, `ConnectorSpecific`, `Inconclusive` или `NotSupported`.

### Влияние на DCAnt

Минимальное необходимое изменение проекта.

### Что проверить дополнительно

Только проверки, без которых вывод остается неполным.

## Ограничения

- Не проводить торговые contract tests на реальном счете.
- Не считать лабораторный результат универсальным для всех connectors.
- Не считать API success подтверждением биржи.
- Не считать задержку подтверждением отмены.
- Не менять production-код во время анализа без отдельного поручения.
- Не объединять исправление архитектуры с самим contract test.
- Не утверждать причины поведения connector, если наблюдается только результат.
