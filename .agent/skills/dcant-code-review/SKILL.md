# DCAnt Code Review

## Назначение

Проводить практический аудит кода DCAnt с приоритетом надежности торгового поведения.

Не оценивать код только по общим архитектурным шаблонам.

Не предлагать глобальную переработку, если проблему можно безопасно исправить локально.

Для DCAnt 1.x учитывать существующую архитектуру и искать минимальные исправления.

Для DCAnt 2.0 проверять соответствие:

- `PROJECT.md`;

- текущему `change.md`;

- утвержденному scope;

- обязательным правилам безопасности.

## Перед началом

1. Найди текущий `change.md`, если он существует.

2. Прочитай только релевантные разделы `PROJECT.md`.

3. Определи, проводится review:

&#x20;  - текущего diff;

&#x20;  - отдельных файлов;

&#x20;  - конкретного дефекта;

&#x20;  - всего небольшого компонента.

4. Не загружай несвязанные документы.

5. Не изменяй код, если пользователь просит только аудит.

## Основные проверки

### 1. Владение ордерами

Проверь:

- доказана ли принадлежность ордера конкретной стратегии;

- используется ли `StrategyInstanceId`;

- используется ли `TradeCycleId`;

- сохраняется ли связь внутренних и broker ID;

- не используется ли только Account и Symbol;

- не используется ли Comment как единственное доказательство;

- не может ли стратегия изменить ручной или чужой ордер.

### 2. Дублирование торговых действий

Проверь возможность появления:

- двух Entry;

- двух ордеров одного GridLevel;

- нескольких логических TP;

- нескольких логических SL;

- повторного EmergencyExit;

- повторной обработки одного Execution;

- повторной отправки effect после timeout.

### 3. Жизненный цикл внешней операции

Проверь:

- не считается ли успешный API-вызов подтверждением биржи;

- не считается ли timeout доказанным отказом;

- выполняется ли reconciliation перед повтором;

- сохраняется ли критический effect до отправки;

- ожидается ли подтверждение отмены перед созданием замены;

- не используется ли фиксированная задержка как доказательство отмены.

### 4. Executions и финансовый учет

Проверь:

- используется ли Execution как фактический торговый факт;

- не рассчитывается ли realized PnL по Last, Bid или Ask;

- не учитывается ли duplicate Execution повторно;

- не отбрасывается ли late Execution;

- корректно ли обрабатываются partial fills;

- рассчитывается ли VWAP по цене и количеству исполнений;

- не смешиваются ли Order и Execution.

### 5. Состояние и конкурентность

Проверь:

- изменяет ли состояние только EngineLoop;

- не изменяют ли его callbacks Quantower;

- не изменяют ли его таймеры напрямую;

- не запускаются ли несколько одинаковых async-операций;

- не остается ли флаг операции навсегда установленным;

- не возникает ли новый TradeCycle до завершения предыдущего;

- не отключаются ли TP и SL из-за ошибки DCA-сетки.

### 6. Persistence и recovery

Если изменение затрагивает persistence, проверь:

- сохраняется ли effect до внешнего вызова;

- атомарны ли связанные записи;

- возможно ли восстановление после restart;

- что произойдет при ошибке записи;

- что произойдет при повреждении или несовместимой версии базы;

- не создается ли пустое состояние при существующей broker position;

- выполняется ли reconciliation до разрешения торговли.

### 7. Quantower Adapter

Проверь:

- не проникают ли типы Quantower в Core;

- копируются ли данные callback в собственные сообщения;

- не принимает ли adapter торговые решения;

- помечены ли неподтвержденные особенности API как Contract Test Requirement;

- не предполагается ли, что `Modify succeeded` означает фактическую замену;

- не предполагается ли гарантированный порядок callbacks.

### 8. Ошибки и диагностика

Проверь:

- отсутствуют ли пустые `catch` в критической логике;

- содержит ли ошибка достаточный контекст;

- записываются ли ID стратегии, цикла, ордера и effect;

- различаются ли факт, предположение и ошибка;

- не скрывается ли ошибка логирования;

- не продолжается ли торговля после критической ошибки конфигурации.

### 9. Контроль сложности

Проверь, не добавлены ли без необходимости:

- новый проект;

- новый архитектурный слой;

- новый интерфейс;

- Generic Repository;

- общий Unit of Work;

- Common project;

- новая таблица;

- новая зависимость;

- универсальная абстракция для одного случая;

- функциональность вне текущего `change.md`.

Не требуй удаления защитного кода только ради уменьшения количества строк.

## Уровни важности

### Критично

Ошибка может привести к:

- лишнему ордеру;

- неправильному объему позиции;

- потере защиты;

- изменению чужого ордера;

- повторному Execution;

- неправильному восстановлению;

- ошибочному PnL;

- началу новой сделки при незавершенной старой.

### Важно

Проблема снижает надежность, тестируемость или сопровождаемость, но не создает немедленный торговый риск.

### Можно позже

Косметическое или необязательное улучшение.

## Формат отчета

### Коротко

Главный вывод в 2–4 предложениях.

### Критично

Для каждого замечания укажи:

- файл и метод;

- конкретную проблему;

- сценарий возникновения;

- возможный результат;

- минимальное исправление.

Если критических проблем нет, напиши: `Не обнаружено`.

### Важно

Тот же формат, но только для значимых некритических проблем.

### Можно позже

Краткий список необязательных улучшений.

### Что проверено

Перечисли фактически просмотренные файлы и сценарии.

### Что нельзя подтвердить

Укажи отсутствующие данные, неподтвержденные особенности Quantower и необходимые contract tests.

## Ограничения

- Не утверждай, что сборка без ошибок доказывает корректность торговли.

- Не утверждай неподтвержденное поведение Quantower или брокера.

- Не изменяй код при запросе только на review.

- Не предлагай полную переработку DCAnt 1.x.

- Не расширяй текущий OpenSpec change.

- Не создавай новые документы автоматически.

- Не называй гипотезу подтвержденным фактом.
