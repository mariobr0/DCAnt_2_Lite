# DCAnt Log Analysis

## Назначение

Восстанавливать фактическую последовательность работы DCAnt по логам и отделять доказанные факты от предположений.

Анализ должен отвечать:

1. Что произошло?
2. В каком порядке?
3. Какое состояние было у бота?
4. Какое действие бот запросил?
5. Что фактически подтвердил Quantower или брокер?
6. Где впервые возникло расхождение?
7. Каких данных не хватает?

## Перед началом

1. Определи временной интервал проблемы.
2. Определи:
   - BotId;
   - StrategyInstanceId;
   - TradeCycleId;
   - account;
   - symbol;
   - direction.
3. Не смешивай логи разных экземпляров стратегии.
4. Проверь timezone и формат времени.
5. Если часть лога отсутствует, скажи об этом прямо.
6. Не восстанавливай отсутствующие события догадкой.

## Нормализация событий

Для каждой записи постарайся извлечь:

```text
Timestamp
Source
EventType
StrategyInstanceId
TradeCycleId
InternalOrderId
ClientRequestId
BrokerOrderId
ExecutionId
OrderRole
GridLevel
Side
Quantity
Price
OrderStatus
PositionQuantity
StateBefore
StateAfter
EffectId
CorrelationId
Message
````

Если поле отсутствует, пометь его как `Unknown`.

## Построение хронологии

Собери последовательность:

```text
Входящее событие
→ локальное состояние до обработки
→ принятое решение
→ созданный effect
→ вызов Quantower API
→ локальный ответ
→ callback или Execution
→ состояние после обработки
```

Не объединяй локальный API response и broker confirmation в одно событие.

## Основные паттерны ошибок

### 1. Дублирующий Entry

Ищи:

* несколько Submit Entry;
* одинаковый ClientRequestId;
* новый Entry после timeout без reconciliation;
* несколько позиций или executions одного intent.

### 2. Дублирующий Grid order

Ищи:

* повторную отправку одного GridLevel;
* повторный callback Filled;
* отсутствие deduplication по order или Execution ID;
* неправильное увеличение next grid level.

### 3. Множественные TP или SL

Ищи:

* `TP not found` при существующем ордере;
* потерю Comment или ClientRequestId после Modify;
* новый broker order ID;
* одновременные вызовы нескольких таймеров;
* повторный `Place` до появления ордера в active orders;
* `Cancel → Delay → Place` без подтверждения отмены;
* несколько полных quantities закрытия.

### 4. Пропавшая защита

Ищи:

* cancel TP или SL без последующего подтвержденного replacement;
* ошибку placement;
* переход состояния, блокирующий retry;
* отключение сопровождения из-за grid error;
* неизвестный order ownership.

### 5. Повторный Execution

Ищи:

* одинаковый ExecutionId;
* одинаковую broker scope;
* повторное увеличение filled quantity;
* повторное обновление PnL;
* поздний Execution после Completed.

### 6. Неправильное восстановление

Ищи:

* запуск торговли до reconciliation;
* новую сделку при существующей позиции;
* неизвестный pending effect;
* локально отсутствующий broker order;
* создание пустого состояния при активной позиции.

### 7. Проблемы конкурентности

Ищи почти одновременные:

* callbacks;
* timer events;
* retry;
* grid fill;
* TP recalculation;
* SL recalculation;
* shutdown;
* reconnect.

Особое внимание уделяй нескольким источникам вызова одного метода.

## Причинно-следственный анализ

Для каждого найденного расхождения укажи:

### Первая подтвержденная аномалия

Самое раннее событие, в котором состояние перестало соответствовать ожидаемому.

### Последствия

Какие последующие ошибки возникли из-за первой аномалии.

### Корневая причина

Указывать только при достаточных доказательствах.

Если доказательств недостаточно, писать:

> Наиболее вероятная причина.

### Альтернативные объяснения

Указать реальные альтернативы, если лог не позволяет их исключить.

### Недостающая диагностика

Назвать конкретные поля или события, которые нужно добавить в лог.

## Пример правильного вывода

Неправильно:

> Binance неправильно модифицирует TP.

Правильно:

> После `Modify succeeded` ранее найденный TP перестал попадать в выборку
> `GetActiveBotOrders("TP")`. Через 902 мс бот выполнил `Place NEW TP`.
>
> Лог не содержит broker order ID и Comment после modify, поэтому нельзя
> доказать, удалил ли старый ордер Binance или измененный ордер потерял
> признак ownership в Quantower.
>
> Для подтверждения нужно записать ID, Comment, quantity и status всех
> активных exit-limit ордеров до и после modify.

## Формат отчета

### Коротко

Главный вывод простыми словами.

### Хронология

Используй последовательный список:

```text
13:15:31.073
Событие:
Состояние:
Действие:
Подтверждение:
```

### Первая аномалия

Точный момент и описание.

### Подтвержденные факты

Только то, что непосредственно видно в логах.

### Наиболее вероятная причина

Объяснение с указанием уровня уверенности:

* высокий;
* средний;
* низкий.

### Альтернативные причины

Только реалистичные варианты.

### Недостающие данные

Каких ID, состояний или callbacks нет в логе.

### Минимальная дополнительная диагностика

Перечень полей и точек логирования.

### Минимальное исправление

Предлагай исправление только если причина достаточно подтверждена.

Если причина не подтверждена, сначала предложи диагностический эксперимент.

## Правила технической точности

* `API success` не равен broker acknowledgement.
* `Order` не равен `Execution`.
* `Timeout` не равен failure.
* Отсутствие записи в логе не доказывает отсутствие события.
* Комментарий ордера не является надежным ownership без подтверждения.
* Одинаковые price и quantity не доказывают duplicate.
* `Thread.Sleep` не доказывает завершение внешней операции.
* Визуальный ордер на графике нужно сопоставлять с ID из API.
* Порядок строк разных потоков может отличаться от причинного порядка, если логирование не синхронизировано.

## Ограничения

* Не изменяй код при запросе только на анализ логов.
* Не объявляй broker bug без contract test.
* Не придумывай отсутствующие значения.
* Не предлагай масштабную переработку до локализации первой аномалии.
* Не смешивай события разных BotId, TradeCycleId или accounts.
* Не скрывай ограничения неполного журнала.

## После создания

Перезапусти Antigravity или открой новый чат, чтобы агент заново обнаружил skills.

Проверить `dcant-log-analysis` можно таким запросом:

```text
Проанализируй приложенный лог DCAnt.

Нужно восстановить хронологию размножения TP, найти первую аномалию
и отделить подтвержденные факты от предположений.
````

Для code review:

```text
Проведи review текущего diff DCAnt.

Не изменяй код. Ищи только ошибки торгового поведения,
конкурентности и избыточную сложность.
```

Для Quantower:

```text
Подготовь один contract test для проверки поведения ModifyOrder
на Binance Futures через Quantower.

Нужно проверить, сохраняются ли broker order ID, Comment и ClientRequestId,
а также остается ли старый ордер активным.
```
