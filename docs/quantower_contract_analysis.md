# Анализ поведения контрактов Quantower API (Этап 7)

В рамках Этапа 7 были проведены проверки (через отражение API и тестовые сборки), а также изучена документация API и локальная база знаний, чтобы подтвердить работу контрактов:

## 1. Синхронные отказы при PlaceOrder
- **Гипотеза:** Если ордер не может быть выставлен (например, `Insufficient Funds`, некорректная цена/объем), API возвращает ошибку синхронно, а не через коллбек.
- **Подтверждено:** Метод `PlaceOrder` возвращает объект `TradingOperationResult`. У этого объекта есть свойство `Status` типа `TradingOperationResultStatus` (со значениями `Success`, `Failure` и др.) и свойство `Message`. 
- **Архитектурное последствие:** `QuantowerAdapter` обязан проверять этот результат прямо во время диспетчеризации `Outbox`. Если `Status == Failure`, адаптер должен сразу отправить `OrderRejected` в `EngineLoop`, чтобы предотвратить зависание стейт-машины (которая иначе будет бесконечно ждать события извне).

## 2. Идентификаторы ордеров (ClientRequestId)
- **Гипотеза:** В Quantower поле `Comment` ордера выступает аналогом `ClientRequestId` и привязывается к ордеру на бирже, если биржа это поддерживает.
- **Ограничения:** Из базы знаний известно, что при использовании `ModifyOrder` (например, для сдвига Тейк-Профита), некоторые коннекторы создают новый ордер и не сохраняют старый `Comment`. 
- **Архитектурное последствие:** Не рекомендуется использовать `ModifyOrder` для изменения цены защитных ордеров без дополнительного слоя корреляции. Безопаснее снять старый ордер (`CancelOrder`) и выставить новый (`PlaceOrder`) с новым `ClientRequestId`. Наш DCAnt бот использует именно подход Cancel-and-Replace, что гарантирует точное отслеживание.

## 3. Rate Limits и Таймауты
- **Гипотеза:** Массовая отправка ордеров может привести к бану по Rate Limits. Отмена ордеров также может зависнуть.
- **Архитектурное решение в DCAnt:** 
  - Мы используем `ActiveGridWindow`, чтобы одновременно выставлять не более заданного числа ордеров (например, 2-3 лимитки), вместо выставления сразу 50 уровней.
  - Буферизация (Debounce) исполнения: мы группируем микро-исполнения (Partial Fills) с окном в 300 мс перед тем, как переставлять тейк-профит. Это кардинально снижает нагрузку на API и защищает от Rate Limits.
  - Ожидающие (Pending) состояния ордеров имеют таймауты (которые будут реализованы в механизме Recovery на Этапе 11).

## Итог
- Механизм `PlaceOrder` защищен от зависаний при нехватке маржи.
- Подход Cancel-and-Replace признан самым безопасным для Quantower API.
