# Анализ контрактов Quantower API

Основываясь на результатах выполнения диагностического плагина `DiagnosticContractTestStrategy` на Binance Futures (через Quantower), были выявлены следующие фактические контракты и поведение API:

## 1. Идемпотентность (Idempotency) при размещении ордеров
- **Ожидание:** Отправка двух ордеров с одинаковым параметром `Comment` (ClientOrderId) приведет к отклонению второго ордера биржей (ошибка дубликата).
- **Фактическое поведение:** Биржа **не отклоняет** второй ордер. Коннектор Quantower успешно размещает оба ордера, присваивая им разные внутренние `OrderId` (GUID). На графике появляются два независимых лимитных ордера.
- **Вывод для архитектуры:** Мы **не можем** полагаться на API биржи/Quantower для обеспечения строгой идемпотентности через параметр `Comment`. DCAnt должен самостоятельно гарантировать отсутствие двойных отправок (через локальный Outbox и машину состояний).

## 2. Асинхронность коллекции Core.Orders
- **Ожидание:** Сразу после синхронного вызова `Core.PlaceOrder` ордер будет доступен в глобальной коллекции `Core.Orders`.
- **Фактическое поведение:** Вызов `PlaceOrder` возвращает `Success` и локальный `OrderId` немедленно, однако глобальная коллекция `Core.Orders` обновляется асинхронно через WebSocket-события от биржи. В нашем тесте ордер №1 был найден в коллекции (так как мы добавили задержку `Thread.Sleep(2000)`), а ордер №2 — нет (так как поиск выполнился мгновенно после его создания).
- **Вывод для архитектуры:** При активной работе нельзя проверять наличие только что отправленного ордера в `Core.Orders` напрямую. Однако, при **восстановлении после сбоя (Recovery)**, эта задержка не играет роли. Бот может безопасно прочитать `Core.Orders`, отфильтровать по `Comment` и найти все ранее выставленные ордера.

## 3. Маппинг полей
- Поле `Comment` в объекте `PlaceOrderRequestParameters` успешно сохраняется и прикрепляется к ордеру.
- Его можно надежно использовать как `ClientOrderId` для связывания локального состояния DCAnt с реальными ордерами на бирже.

## 4. Модификация ордеров (ModifyOrder)
- **Ожидание:** При изменении цены ордера через `Core.ModifyOrder` биржа может выполнить операцию `Cancel/Replace`, что потенциально приведет к изменениию внутреннего `OrderId` и потере поля `Comment`.
- **Фактическое поведение:** При вызове `Core.ModifyOrder(..., price: newPrice)` ордер успешно обновляет цену. При этом его внутренний `Id` в Quantower **остается неизменным**, а поле `Comment` **сохраняется в первозданном виде**.
- **Вывод для архитектуры:** DCA-бот может безопасно сдвигать лимитные ордера (трейлинг сетки, подтягивание тейк-профитов), просто вызывая `ModifyOrder`. Нам не нужно реализовывать сложную логику перепривязки ID ордеров в базе данных после каждого изменения цены.

## Архитектурное решение для DCAnt
1. **Normal Flow**: При нормальной работе DCAnt использует `Outbox` для гарантированной отправки ордера ровно один раз. Отправленный ордер отслеживается через подписку на события `OrderAdded` и `OrderUpdated`.
2. **Recovery Flow**: При перезапуске DCAnt загружает свои незавершенные `Saga/ProcessManager` стейты, берет ожидаемый `Comment`, и ищет его в `Core.Orders` (который к моменту инициализации плагина уже заполнен коннектором).
    - Если ордер найден — стейт актуализируется из биржи.
    - Если ордер не найден (и не найден в истории) — значит сбой произошел до того, как биржа его приняла, и отправку можно безопасно повторить.
