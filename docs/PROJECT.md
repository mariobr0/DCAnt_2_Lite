# DCAnt 2.0

## 1. Назначение проекта

DCAnt 2.0 является локальной алгоритмической DCA-стратегией для торгового терминала Quantower.

Главная цель проекта состоит в создании надежного и предсказуемого торгового бота, который:

- открывает позицию по заданным условиям;
- размещает DCA-ордера;
- корректно обрабатывает частичные исполнения;
- рассчитывает фактический объем и среднюю цену позиции;
- поддерживает Take Profit и Stop Loss;
- не создает дублирующие ордера;
- восстанавливается после перезапуска;
- сверяет локальное состояние с фактическим состоянием брокера;
- ведет статистику по фактическим исполнениям.

DCAnt 2.0 не является:

- универсальной торговой платформой;
- системой управления инвестиционным портфелем;
- микросервисной системой;
- HFT-движком;
- внешним торговым сервером;
- заменой Quantower;
- фреймворком для любых видов торговых стратегий.

Архитектура должна соответствовать масштабу одного локального плагина Quantower.

## 2. Принцип разработки

Проект развивается небольшими законченными изменениями.

Для каждой значимой задачи создается один файл:

`openspec/changes/<change-name>/change.md`

Каждый `change.md` объясняет:

1. Зачем нужно изменение.
2. Как бот должен вести себя после изменения.
3. Что не входит в задачу.
4. Какие действия необходимо выполнить.
5. Как проверить результат.

Реализация начинается только после просмотра `change.md`.

Проект не должен усложняться ради гипотетических будущих функций.

Новая абстракция, библиотека, сборка или таблица добавляется только тогда, когда решает конкретную текущую проблему.

## 3. Объем первой версии

Первая рабочая версия поддерживает:

- один `StrategyRuntime` на экземпляр стратегии;
- один торговый счет;
- один инструмент;
- один активный `TradeCycle`;
- Long-направление;
- USDT linear instruments;
- Market Entry;
- DCA-сетку;
- один логический Take Profit;
- один логический Stop Loss;
- частичные исполнения;
- безопасную отмену и замену ордеров;
- локальную базу SQLite;
- Outbox для исходящих торговых намерений;
- восстановление после перезапуска;
- reconciliation с Quantower;
- базовое логирование;
- базовый экспорт статистики.

В первую версию не входят:

- одновременные Long и Short;
- Hedge Mode;
- Netting coordination;
- несколько торговых счетов в одном runtime;
- несколько инструментов в одном runtime;
- управление общим капиталом нескольких стратегий;
- multi-currency accounting;
- распределение funding между стратегиями;
- полный Event Sourcing;
- persistent Inbox;
- универсальные State Snapshots;
- отдельное глобальное хранилище Scheduled Deadlines;
- машинное обучение;
- портфельная аналитика;
- сложная визуальная панель;
- автоматические UI-тесты Quantower.

Отложенные функции могут быть добавлены позже отдельными изменениями.

## 4. Структура проекта

DCAnt 2.0 использует следующую структуру:

- `.agents/rules/` содержит постоянные правила поведения AI-агента.
- `.agents/skills/` содержит специализированные инструкции для AI-агента.
- `openspec/` содержит облегченный процесс OpenSpec.
- `src/` содержит production-код.
- `tests/` содержит автоматические тесты.
- `PROJECT.md` содержит постоянную карту проекта.
- `Directory.Build.props` содержит общие настройки компиляции.
- `.editorconfig` содержит единые правила оформления кода.

Проекты solution:

- `DCAnt2.Core`;
- `DCAnt2.Infrastructure`;
- `DCAnt2.Quantower`;
- `DCAnt2.Plugin`;
- `DCAnt2.Tests`;
- `DCAnt2.QuantowerTests`.

## 5. Обязательные правила безопасности (сокращенный раздел)

- **Только EngineLoop изменяет торговое состояние.** События Quantower, таймеры и UI-команды передаются в EngineLoop в виде сообщений и не изменяют состояние напрямую.
- **Бот управляет только своими ордерами.** Принадлежность ордера должна подтверждаться внутренними идентификаторами. Совпадения только по счету, инструменту или комментарию недостаточно.
- **Тайм-аут не считается отказом операции.** Если результат отправки, отмены или изменения ордера неизвестен, бот сначала сверяется с брокером и не повторяет команду вслепую.
- **Торговое намерение сохраняется до отправки.** Перед критическим вызовом Quantower бот сохраняет действие в Outbox, чтобы после сбоя не забыть о попытке и не создать дублирующий ордер.
- **Фактический учет строится по Executions.** Объем позиции, VWAP, комиссии и realized PnL рассчитываются по фактическим исполнениям, а не по `Last`, `Bid`, `Ask` или одному статусу ордера.
- **Повторные события обрабатываются безопасно.** Повторный Execution или order event не должен повторно изменять позицию, статистику или состояние ордера. Поздние Executions нельзя молча отбрасывать.
- **Неоднозначность запрещает увеличение риска.** Если состояние позиции или ордера нельзя установить достоверно, бот прекращает новые входы и DCA, сохраняет подтвержденную защиту и запускает reconciliation.
- **Защита позиции важнее DCA-сетки.** Ошибка сетки не должна отключать Take Profit, Stop Loss, обработку Executions, reconciliation или возможность закрыть позицию.
- **Замена защитного ордера требует подтверждения.** Новый TP или SL нельзя создавать до подтвержденной отмены или доказанного отсутствия предыдущего ордера. Фиксированная задержка не является подтверждением.
- **Новый TradeCycle начинается только после завершения предыдущего.** Отсутствие позиции и связанных ордеров должно быть подтверждено. Отправка команды закрытия сама по себе не означает, что сделка завершена.
- **Критические ошибки не скрываются.** Ошибки торговых операций, persistence и reconciliation должны записываться в лог. Пустые `catch` в критической логике запрещены.
- **Неподтвержденное поведение Quantower не принимается за гарантию.** Особенности `ModifyOrder`, `Cancel`, `ClientRequestId`, partial fills и порядка событий проверяются отдельными contract tests на демо-счете.

## 6. Использование облегченного OpenSpec

Для значимого изменения создается:

`openspec/changes/<change-name>/change.md`

Структура файла:

- название;
- зачем нужно изменение;
- требуемое поведение;
- что не входит;
- задачи;
- результат проверки.

OpenSpec change нужен для:

- нового торгового поведения;
- изменения ордеров;
- DCA-математики;
- TP и SL;
- persistence;
- recovery;
- новых пользовательских настроек;
- важного исправления торговой логики.

OpenSpec change не нужен для:

- форматирования;
- исправления опечатки;
- удаления неиспользуемого `using`;
- изменения комментария;
- исправления ссылки;
- локального рефакторинга без изменения поведения;
- добавления диагностического лога.

## 7. Рабочий цикл

Работа над значимым изменением проходит в следующем порядке:

1. Формулируется идея.
2. Создается `change.md`.
3. Владелец проекта проверяет `change.md`.
4. Агент реализует только утвержденный scope.
5. Выполняется Release-сборка.
6. Выполняются применимые тесты.
7. Реализация сравнивается с требуемым поведением.
8. Заполняется результат проверки.
9. Change архивируется.
10. Изменения сохраняются отдельным Git-коммитом.

Основные команды:

- `/opsx-propose`;
- `/opsx-apply`;
- `/opsx-verify`;
- `/opsx-archive`.

Код не считается правильным только потому, что он компилируется.

Сборка проверяет техническую корректность C#, но не доказывает корректность торгового поведения.
